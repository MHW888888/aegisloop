# AegisLoop Growth Checklist

Use this as a lightweight launch plan. The goal is not spam. The goal is to make the project easy to understand, easy to trust, and easy to share.

## Already Done

- Public GitHub repository
- MIT license
- README with quick start and safety model
- Architecture diagram
- Sanitized extension screenshot
- GitHub Actions syntax and fixture checks
- Release notes through the current v0.3.x hardening line
- Lite onboarding docs
- Parallel Safe Mode docs
- Dual Briefing templates
- Issue and PR templates
- Contributing guide
- First-run and troubleshooting docs
- Good first issues for contributors
- Tester guide and compatibility issue packs
- Volunteer recruitment issue

## Next High-Impact Steps

1. Pin the repository on the GitHub profile.
2. Keep one sanitized screenshot near the top of the README.
3. Add a short GIF when the extension states are stable.
4. Review incoming PRs quickly, especially docs/demo PRs.
5. Close completed or stale issues so new visitors see a clean queue.
6. Post the short launch note on X, LinkedIn, V2EX, Hacker News "Show HN", Reddit, Xiaohongshu, or a developer WeChat group.
7. Ask 3-5 technical friends to read the README and tell you where they get confused.
8. Turn the clearest confusion into a README improvement.
9. Keep 3-5 small "good first issue" items open for contributors.
10. Reply quickly to first-time contributors and close completed issues cleanly.
11. Use [demo-issue-pack.md](demo-issue-pack.md) to split visual demo work into contributor-friendly tasks.
12. Use [promotion-playbook.md](promotion-playbook.md) for weekly outreach.

## Suggested One-Liner

AegisLoop is a guarded local bridge that lets ChatGPT plan and Codex execute, with explicit arming, workspace locks, Run Capsules, ACK/NACK, turn tokens, and audit logs.

## 中文一句话

AegisLoop 是一个 ChatGPT x Codex 本地自动化桥：ChatGPT 负责规划，Codex 负责本地执行，中间加显式授权、工作区锁、Run Capsule、ACK/NACK、turn token 和审计日志。

## Weekly Routine

- Check open issues and reply to contributors.
- Review or close incoming PRs within 24-48 hours when possible.
- Keep the README screenshot and version notes reasonably fresh.
- Convert repeated questions into docs.
- Share only real progress: releases, docs, demo GIFs, contributor PRs, and lessons learned.
- Do not buy stars, spam communities, or overclaim capabilities.

## Healthy Signals

- A first-time contributor can find a task in under 2 minutes.
- A tester can report one browser/model path in 5-10 minutes.
- Issues have clear "done" criteria and privacy rules.
- PRs receive a clear review or maintainer response.
- Public posts point to a concrete ask, not vague hype.
